## 数据架构

### 存储策略
```
localStorage（主） → 读写即时
CloudBase（备份）  → Web SDK 直接数据库访问（匿名登录）

同步机制（2026-06-18 更新）：
- 同步通道: CloudBase Web SDK 直接数据库访问（不经过云函数）
- 依赖安全规则: 业务集合开放 read:true, write:"auth != null"
- dbProxy 云函数保留作为备用通道（当前不使用）
- parents / email_codes 保持仅 owner 可读写（保护隐私）
```

### 数据集合（10 个）

|------|------|

### 验证码邮件配置

**架构（2026-06-17 更新）：**
- 前端不再通过 SDK `callFunction` 调用（浏览器存在兼容问题），改为 `fetch()` HTTP 直连
- CloudBase HTTP 访问服务 → Event 函数，公开 URL 无需登录态
- 前端 `js/login.js` → `callVerifyCode()` 函数封装

**首次配置：**
1. QQ 邮箱 → 设置 → 账户 → POP3/SMTP → 开启 → 生成授权码
2. CloudBase 控制台 → 云函数 verify-code → 环境变量：
   - `EMAIL_USER` = QQ 邮箱地址
   - `EMAIL_PASS` = 授权码（非 QQ 密码）
3. `email_codes` 集合由云函数自动创建（不再需要手动建表）

---

### 验证码故障排查
```bash
# 1. 检查云函数环境变量
npx tcb fn invoke verify-code --params '{"action":"ping"}'
# → 应返回 hasEmailUser:true, hasEmailPass:true

# 2. 查看最近调用日志
npx tcb fn log verify-code --limit 20

# 3. 测试发送（CLI 调用，绕过前端）
npx tcb fn invoke verify-code --params '{"action":"send","email":"你的邮箱@qq.com"}'

# 4. 常见错误：
# "邮箱授权码无效" → QQ 授权码过期，去 QQ 邮箱重新生成
# "邮件服务器连接失败" → SMTP 网络不通，稍后重试
# "发送过于频繁" → 触发限流，等一会再试
# "email_codes collection not exist" → 云函数自动创建失败
```

- `css/style.css` 被 7 个页面加载（公开4页 + login + teacher + parent）
